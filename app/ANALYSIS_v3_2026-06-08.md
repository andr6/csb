> Date: 2026-06-08 | Commit: c92cb19 | Effort: E5 | 14,937 lines of JS across 54 files

## TL;DR

CSB has evolved from a fun prototype into a **surprisingly mature Express backend** with DI factories, structured logging, provider plugins, and atomic SQLite limits. But it's still haunted by its prototype DNA: **unauthenticated tournament endpoints**, a **test-mode auth bypass**, **event-loop-blocking analytics**, and **frontend modules that leak global state**. The good news: the hard architectural work is done. The bad news: the sharp edges will cut someone.

---

## 1. THE GOOD — What's Actually Well Done

### 1.1 Route Factory DI Pattern
`createFireRouter(deps)`, `createJudgeRouter(deps)`, `createAuthRouter(deps)` — every route factory accepts dependency overrides. This is **production-grade testability**. You can inject mock `callContestant`, mock `validatePageToken`, mock `dailyTryIncrement`. Most Node.js projects never get this right.

*Files: `routes/*.js` — all router factories*

### 1.2 Security Headers
CSP, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, disabled `x-powered-by`. Someone read OWASP and applied it.

*File: `app.js:203-213`*

### 1.3 PKCE + Signed JWT OAuth State
No server-side state storage. The OAuth state is a signed JWT encoding provider + code_verifier. This survives PM2 cluster mode, restarts, and multi-instance deployments. Elegant.

*File: `lib/oauthService.js:29-100`*

### 1.4 Atomic SQLite Daily Limits
`dailyTryIncrement` uses `INSERT OR IGNORE` + `UPDATE ... SET count = count + 1` in a single statement. Race-safe, restart-safe, multi-process-safe. The fallback to permissive mode on SQLite failure is the right call — fail open on limit enforcement, not fail closed.

*File: `lib/dailyLimits.js:44-96`*

### 1.5 JSON Repair Pipeline with Timeout Guard
Four-stage repair (raw parse → JSON5 normalizer → regex fix → brace close) with a 500ms `setTimeout` abort. This is pragmatic engineering for unreliable LLM JSON output.

*File: `lib/judge.js:112-178`*

### 1.6 Model Health Tracking
`model_health_log` table + `getHealthyModelIds()` filters out models with <60% success rate. Automatic degradation without human intervention.

*File: `lib/modelHealth.js`*

### 1.7 Provider Plugin Registry
`registerProvider(name, handler)` → `getProvider(name)` → `dispatch`. Clean extension point. Adding a new provider means one `registerProvider` call.

*File: `lib/providers.js:17-45`*

### 1.8 Analytics Batch Endpoint
`POST /api/analytics/batch` replaces 22 parallel frontend fetches with one round-trip. Smart optimization.

*File: `routes/analytics.js:109-155`*

### 1.9 Graceful Shutdown
SIGTERM handler saves metrics, flushes webhook queue, closes DB, then exits. With one caveat (see Bad).

*File: `server.js:47-92`*

### 1.10 Wilson Score Interval for Blind Alignment
Not just `aligned / total` — computes 95% confidence intervals. Statistical rigor in a joke app.

*File: `lib/analyticsEngine.js:455-476`*

## 2. THE BAD — What's Broken, Weak, or Dangerous

### 2.1 🔴 CRITICAL: Tournament Endpoints Completely Unauthenticated
**Any visitor** can `POST /api/tournament` to create a tournament, `POST /api/tournament/:id/run-match` to burn API tokens, and `POST /api/tournament/:id/advance` to manipulate brackets. No `x-page-token`. No auth. No daily limit. This is a **free LLM API proxy** for attackers.

*File: `routes/tournament.js` — entire file lacks auth middleware*

**So what:** Add `authMw.requireAuth` + `dailyTryIncrement` + `validatePageToken` to all tournament mutation endpoints. Treat `run-match` as a `fire` equivalent.

### 2.2 🔴 CRITICAL: Test-Mode Auth Bypass
`app.js:121-123`:
```js
if (process.env.NODE_ENV === "test") {
  authMw.requireAuth = function(req, res, next) { next(); };
}
```
If `NODE_ENV=test` leaks to production (easy with PM2 misconfiguration), **all auth is disabled globally**.

**So what:** Remove the bypass. Use explicit `deps` injection in tests instead.

### 2.3 🔴 CRITICAL: `SESSION_SECRET` Defaults to Empty String
`lib/oauthService.js:5` — `SESSION_SECRET` falls back to empty string (`process.env.SESSION_SECRET || ""`). If unset, the OAuth state JWT is **signed with an empty key**. An attacker can forge a valid state JWT, bypass CSRF protection, and potentially perform account linking attacks.

**So what:** Fail startup if `SESSION_SECRET` is missing or shorter than 32 bytes.

### 2.4 🔴 CRITICAL: `PAGE_TOKEN_SECRET` Defaults to Empty String
`lib/fireHelpers.js:15` — `PAGE_TOKEN_SECRET` falls back to empty string. If combined with empty `ALLOWED_ORIGINS`, an attacker can forge page tokens and bypass CSRF-like protection on `/api/fire` and `/api/blind-mapping`.

**So what:** Same as above — fail startup if `PAGE_TOKEN_SECRET` is missing or too short.

### 2.5 🔴 HIGH: Page Token Secret Non-Deterministic (Even When Set)
`PAGE_TOKEN_SECRET` falls back to `crypto.randomBytes(32)` at module load (`app.js:60`). Every restart invalidates all issued page tokens. Deploy → every active user's tokens break.

**So what:** Require `PAGE_TOKEN_SECRET` in env. Fail startup if missing.

### 2.4 🔴 HIGH: Password Reset Secret Non-Deterministic
`lib/oauthService.js:445` — reset token HMAC uses `process.env.RESET_SECRET || process.env.PAGE_TOKEN_SECRET || crypto.randomBytes(32)`. Same problem: restart invalidates in-flight reset emails.

**So what:** Same fix — fail startup if `RESET_SECRET` is missing.

### 2.5 🔴 HIGH: Tournament `run-match` Fires Contestants Sequentially
`routes/tournament.js:170-228` calls `execModels(slotA)` then `execModels(slotB)` sequentially. A 16-model tournament with 60s timeouts per contestant = **32 minutes of sequential wall-clock time**.

**So what:** Fire contestants in parallel with `Promise.all`.

### 2.6 🔴 HIGH: Analytics Engine Blocks the Event Loop
`lib/analyticsEngine.js` — all functions are synchronous, in-memory, and process entire datasets. On a production database with 10K runs, `computePromptTopics` + `computeWinStreaks` + `computeBlindAlignment` + 15 other functions will block the event loop for **multiple seconds**. Node's single thread means all other requests stall.

**So what:** Move analytics computation to a worker thread (`worker_threads`), or precompute hourly rollups and serve from cache.

### 2.7 🔴 HIGH: Uncached Synchronous File Reads on Every Request
`routes/fire.js:50,62` reads `pack-prompts.json` and `mode-prompts.json` with `fs.readFileSync` on **every request**. These files are 5KB and 200KB. Under load, this is pure I/O waste.

**So what:** Load at startup, cache in memory. These files don't change at runtime.

### 2.8 🔴 MEDIUM: `willNotify` Logic Bug in Judge Route
`routes/judge.js:117`:
```js
var willNotify = deps.notifyWebhook !== undefined;
```
But `notifyWebhookFn` resolves from `deps.notifyWebhook || _notifyWebhook`. If `WEBHOOK_URL` is configured but `deps.notifyWebhook` is undefined (the common case), `willNotify` is `false` and the crown-change webhook is **never sent**.

**So what:** Check `notifyWebhookFn` truthiness, not `deps.notifyWebhook !== undefined`.

### 2.9 🔴 MEDIUM: No Size Limit on Judge Prompt Input
`routes/judge.js` accepts `req.body.responses` with no length validation before embedding into the judge prompt. A malicious client could submit 50MB of responses, inflating the judge API call to hundreds of dollars.

**So what:** Truncate responses to a max token budget before building the judge prompt.

### 2.10 🔴 MEDIUM: No Concurrency Control on Tournament State
`routes/tournament.js` modifies `_tournaments` in-memory state with no locks. Two simultaneous `advanceWinner` or `run-match` requests race and corrupt bracket state.

**So what:** Use SQLite as the single source of truth with `BEGIN IMMEDIATE` transactions, not in-memory mutable state.

### 2.11 🔴 MEDIUM: Mock OTP Transports Log Secrets to Console
`app.js:305-320` — `sendEmailOtp` and `sendSmsOtp` fallbacks log OTP codes and reset links to stdout. In containerized environments, stdout goes to log aggregation. **Secrets in logs**.

**So what:** Never log OTPs. Log metadata only (`OTP sent to masked email`).

### 2.12 🔴 MEDIUM: `jsonMode` Forced on All Providers
`lib/providers.js:371` passes `jsonMode: true` to all providers. Anthropic and OpenAI set `response_format: { type: "json_object" }`, but not all models support this. Causes hard API errors for unsupported models.

**So what:** Only set `jsonMode` for providers/models known to support it.

### 2.13 🔴 LOW: Health Probes Cost Money
`checkProviderHealth` for Anthropic and OpenAI makes **actual POST requests** to score health. Every health check consumes tokens and costs money.

**So what:** Use cheap `GET /models` or `HEAD` endpoints for health probes.

### 2.14 🔴 LOW: `VOICE_CACHE` Unbounded Growth
`lib/models.js:8` — `var VOICE_CACHE = {}` has no eviction, no TTL. In a long-running process, this leaks memory indefinitely.

**So what:** Use an LRU cache with a 100-entry limit.

### 2.15 🔴 LOW: Analytics Cache Keys Are Non-Deterministic
`routes/analytics.js:85` — cache key is `JSON.stringify(filters)`. If `buildRunFilters` returns objects with different key ordering, cache hits fail silently.

**So what:** Sort keys before stringifying, or use a stable hash.

### 2.16 🔴 MEDIUM: Dead Module `leaderboardRepository.js`
`lib/repositories/leaderboardRepository.js` (116 lines) references the `leaderboard_entries` table dropped in migration 007. It is **never imported anywhere** in the application. Dead code that will crash if accidentally imported.

**So what:** Delete the file. It serves no purpose.

### 2.17 🔴 MEDIUM: String-Concatenated SQL in Multiple Files
Despite widespread parameterized queries, several files still concatenate SQL strings:
- `lib/repositories/leaderboardRepository.js:52-58` — `escapeSqlString` instead of parameterized queries
- `lib/metrics.js:103-104,144-145` — timestamp concatenation into SQL
- `lib/history.js:23-26` — raw SQL string construction

**So what:** Standardize on parameterized queries. Remove raw SQL APIs from `sqlite.js` to prevent regression.

### 2.18 🔴 LOW: Migration Schema Drift Risk
Migration 021 creates `model_health_log`, but `lib/modelHealth.js:6-21` also auto-creates the same table in `_ensureTable()`. If the migration is modified but `_ensureTable` is not updated, fresh deployments get divergent schemas depending on import order.

**So what:** Remove `_ensureTable` from modules. Migrations are the single source of truth for schema.

### 2.19 🔴 HIGH: OAuth `redirect_uri` Built from Attacker-Controlled Header
`routes/auth.js:558-563` — `getOAuthRedirectUri` uses `req.headers.origin` as a fallback when `OAUTH_REDIRECT_BASE` is unset. The `Origin` header is **fully attacker-controlled**. Allows redirecting the OAuth authorization code to an attacker-controlled domain.

**So what:** Remove the fallback. Use a strict allowlist or single hardcoded `OAUTH_REDIRECT_BASE`. Fail startup if unset.

### 2.20 🔴 HIGH: OAuth Callback `postMessage` Uses Wildcard TargetOrigin
`routes/auth.js:672` — `window.opener.postMessage(..., "*")`. Any domain that opens the OAuth popup can receive the user's session token. Enables token theft via malicious opener pages.

**So what:** Restrict `targetOrigin` to the exact expected origin. Never use `"*"`.

### 2.21 🔴 HIGH: PKCE `code_verifier` Stored in Unencrypted JWT State
`lib/oauthService.js:55-62` — The state JWT contains the plaintext PKCE `code_verifier` in the payload. The JWT is signed but **not encrypted**. Since the state parameter travels in URLs (browser history, server logs, referrer headers), the code verifier is exposed to URL leaks.

**So what:** Store the verifier server-side (even in a signed cookie) or encrypt the JWT payload before including it in URLs.

### 2.22 🔴 MEDIUM: `escapeSqlString` Uses Naive Quote-Doubling
`lib/sqlite.js:72-74` — `escapeSqlString` doubles single quotes but does **not** handle backslash escapes or other injection vectors. Exported and used in `leaderboardRepository.js`. Creates a false sense of security.

**So what:** Remove `escapeSqlString`. Use parameterized queries exclusively.

### 2.23 🔴 MEDIUM: `backup.js` Uses Weak SQL Escaping for `VACUUM INTO` Path
`lib/backup.js:57` — `runSql("VACUUM INTO '" + destPath.replace(/'/g, "''") + "';")`. Path traversal possible if `CSB_BACKUPS_DIR` is attacker-controlled.

**So what:** Validate backup path is within expected directory. Use parameterized query if possible, or sanitize aggressively.

### 2.24 🔴 MEDIUM: No Rollback Migrations
All 21 migrations are **up-only**. No `down/` directory, no rollback function. If a migration fails halfway through (e.g., 001_initial_schema.sql with multiple CREATE TABLE statements), the database is left in a partially migrated state because `applyMigration` splits on `;\s*\n` and runs statements individually **without `BEGIN ... COMMIT`**.

**So what:** Wrap multi-statement migrations in transactions. Add down-migrations for production safety.

### 2.20 🔴 LOW: Duplicate / Shadowed Functions
- `lib/fireHelpers.js:5-21` defines `validatePageToken`, which duplicates the identical function in `app.js:63-82`. The app.js version is the one actually used.
- `lib/fireHelpers.js:23-30` defines `isDisplayableLeaderboardAnswer`, which duplicates the same function in `lib/leaderboard.js:1-8`.

**So what:** Deduplicate. One source of truth per function.

## 3. WHAT DOESN'T MAKE SENSE — Nonsensical Patterns

### 3.1 Factory That Mutates Global State
`createApp` returns an Express app but also calls `seedAdminUser()` (async side effect) and `runBackup()` (sync side effect) before returning. A factory should **produce** objects, not **mutate** the universe.

*File: `app.js:131-179, 408`*

### 3.2 `requireAdminAuth` Comment Contradicts Implementation
The comment on line 218 says `requireAdminAuth` must **never** be chained with `authMw.requireAuth`. But both are injected from the same composition root and used together in `routes/analytics.js:12`. The comment is a lie.

### 3.3 `server.js` dotenv Override Logic Is Backwards
```js
dotenv.config({ override: process.env.NODE_ENV !== "test" });
```
When `NODE_ENV` is undefined (common in production), `override` is `true`. The `.env` file **overrides system environment variables**. This is the opposite of safe production behavior.

*File: `server.js:2`*

### 3.4 OAuth `renderOAuthResult` Double-Stringifies
`routes/auth.js:672`:
```js
var payload = JSON.stringify(JSON.stringify({ success: true, token: jwtToken }));
```
This produces a JSON-string-encoded-as-a-JavaScript-string-literal. `window.opener.postMessage` receives a string, not an object. The frontend probably has to `JSON.parse` twice. This is either a bug or a very strange contract.

### 3.5 `computeWinStreaks` Iterates Rows Twice for No Reason
Lines 433 and 439 both iterate `rows`. Could be done in a single pass. Not a bug, just waste.

### 3.6 `MODEL_CATALOGUE` Iterates ALL Environment Variables
`lib/config.js:125-136` loops `Object.keys(process.env)` with a regex. This is O(n) over **all** environment variables — including `PATH`, `HOME`, `SSH_AGENT_LAUNCHER`, etc. On systems with 200+ env vars, this is unnecessary overhead.

### 3.7 Tournament Module Requires 13 Direct Dependencies
`routes/tournament.js` requires `lib/config`, `lib/models`, `lib/providers`, `lib/judge`, `lib/packs`, `lib/dailyLimits`, `lib/sqlite`, `lib/analysisRuns`, `lib/history`, `lib/metrics`, `lib/webhook`, `lib/tournament`, `lib/validation`. The **router knows about 13 subsystems**. This is the opposite of the DI pattern used elsewhere.

### 3.8 `compatibleModes` Includes "custom" But Custom Mode Was Deleted
`lib/packs.js:14` — every pack lists `"custom"` in `compatibleModes`, but custom mode was removed from the UI in the last refactor. The backend still advertises it. Dead parameter.

### 3.9 Live Site Config Returns Zero Modes and Zero Providers
```bash
curl -s https://www.chatshitbob.com/api/config | jq '{models: .models | length, packs: .packs | length, modes: .modes | length, providerCount: .providers | length}'
# {"models": 8, "packs": 9, "modes": 0, "providerCount": 0}
```
The config endpoint is broken in production. Modes and providers arrays are empty. This means the frontend falls back to hardcoded defaults, which defeats the purpose of backend-owned config.

### 3.10 `addHistoryEntry` Is a No-Op Stub
`lib/history.js:17-19`:
```js
function addHistoryEntry(_entry) {
  return readHistory();
}
```
This function is exported but does nothing. `analysis_runs` replaced history, but the stub remains.

## 4. ENHANCEMENTS — Existing Function Improvements

### 4.1 Tournament Mode: Make It Actually Usable
**Current state:** Backend has full bracket computation, auto-run match endpoint, and SQLite persistence. Frontend has a tournament module that likely doesn't fully leverage it.

**Enhancements:**
- Add real-time bracket visualization (SVG or CSS grid)
- Show match history with clickable replays (link to the `analysisRun` record)
- Add "spectator mode" — users can watch tournaments progress without paying for API calls
- Add tournament seeding options (random, Elo-based, manual)
- Export tournament results as a shareable image/card

### 4.2 Security Mode: Make It Actually Test Security
**Current state:** "Redteam" mode tests for system-prompt leakage, jailbreaks, and over-refusal. But the prompts are generic and the scoring is just another judge persona.

**Enhancements:**
- Add **actual jailbreak attempt detection** — pattern-matching for common jailbreak prefixes ("DAN", "Developer Mode", "ignore previous instructions")
- Score **refusal consistency** — does the model refuse the same prompt every time, or is it stochastic?
- Add **prompt injection detection** — embed hidden instructions and check if the model follows them
- Generate a **security report card** per model (vulnerability categories, severity ratings)
- Add **fuzz testing** — mutate prompts slightly and measure response stability

### 4.3 Analytics Dashboard: Precompute and Visualize
**Current state:** Analytics are computed on-demand, blocking the event loop. The dashboard has 20+ panels but no caching strategy.

**Enhancements:**
- Hourly rollup cron job (already partially implemented via `startHourlyRollup`)
- Add **trend graphs** — line charts showing model performance over time
- Add **A/B comparison** — select two models, compare head-to-head across all metrics
- Add **cost-per-answer tracking** — real spend per model based on token counts
- Add **export to CSV/JSON** for data nerds

### 4.4 Pack System: Dynamic Pack Builder
**Current state:** 9 hardcoded packs in `lib/packs.js`. Adding a pack requires a code deploy.

**Enhancements:**
- Admin UI to create packs (persona name, judge prompt, character base, provider flavours)
- Community pack sharing (pack marketplace)
- Pack rating/voting system
- A/B test packs against each other

### 4.5 Judge System: Multi-Judge Consensus
**Current state:** `JUDGE_RUNS` allows 1-5 judge runs with median scoring. But all judges use the same model.

**Enhancements:**
- **Multi-judge panel** — use different judge models (Claude, GPT-4o, Gemini) and compute inter-rater reliability
- **Weighted scoring** — let users weight criteria (creativity vs accuracy vs brevity)
- **Judge bias detection** — does Claude consistently score OpenAI models lower? Track and expose judge bias.
- **Human-in-the-loop appeals** — users can flag disagreeable scores, triggering a re-judge or community vote

### 4.6 Blind Mapping: Make It Truly Blind
**Current state:** Blind mapping shuffles model IDs, but the UI still shows provider colors and the leaderboard exposes identities.

**Enhancements:**
- **True blind mode** — hide all model identities until the user votes
- **Reveal animation** — dramatic unmasking after voting
- **Blind leaderboard** — separate scoring for "human guess accuracy" vs "model quality"

## 8. VERDICT

CSB is a **B+ concept with C+ execution** — and that's not an insult. Most side projects never get DI factories, atomic SQLite limits, or Wilson score intervals. The foundation is solid.

The gap is between "works on my machine" and "survives the internet." The **empty `SESSION_SECRET` and `PAGE_TOKEN_SECRET` fallbacks** mean a default deployment has **no CSRF protection at all**. The tournament auth gap is a free API proxy. The OAuth `redirect_uri` fallback and wildcard `postMessage` are exploitable by any malicious page. These are not edge cases — they are **default-configuration vulnerabilities**.

Fix Phase 1 this week. The rest can wait. But don't wait long — an unauthenticated tournament endpoint plus empty secrets means anyone on the internet can burn your API keys with zero effort.
